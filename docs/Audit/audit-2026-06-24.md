## Audit 1 — Dependensi Antar Logika

### 1.1 Alur Data Utama (Verified — Konsisten)

```
page.tsx (Server Component)
  → requireRole("contractor")         [src/lib/auth/session.ts]
  → *-service.ts (server-only)        [src/features/*/]
    → db.query.* / db.select()        [src/lib/db/index.ts → Neon]
  → passes typed props → _components/*.tsx

Form Submit (Client)
  → Server Action (actions.ts)        [src/features/*/actions.ts]
    → Zod validate (schemas.ts)
    → *-service.ts (DB mutation)
    → revalidatePath()
    → returns { success, error }
```

Alur ini **konsisten** di seluruh 8 fitur aktif. Tidak ada penyimpangan pola.

### 1.2 Dependensi Kritis yang Perlu Diperhatikan

**`wbs-service.ts` → `projects` tabel (side effect tersembunyi)**

Setiap mutasi WBS item memanggil `recalculateProjectProgress()` secara internal, yang meng-UPDATE kolom `projects.progress`. Side effect ini tidak terdokumentasi di Chain of Truth. Jika ada service lain yang juga mengupdate `projects.progress` secara langsung di masa depan, berpotensi terjadi konflik nilai.

**`deleteProject` → transaksi manual (tanpa FK CASCADE)**

`src/features/projects/projects-service.ts` menggunakan DB transaction manual untuk menghapus semua tabel relasi (`wbsItems`, `dailyReports`, `projectMembers`, `materials`, `materialUsages`, `projectPhotos`). Karena tidak ada FK constraint di schema Drizzle, fungsi ini **WAJIB diperbarui secara manual** setiap kali tabel baru dengan relasi ke `projects` ditambahkan. Tidak ada safety net otomatis.

**`dashboard-service.ts` → `materials` + `projectPhotos` (dependensi baru pasca Tahap 5)**

Sejak implementasi Tahap 5 (2026-06-24 04:30), `getDashboardData` membaca tabel nyata untuk `materialsRecordedTotal` dan `photosToday`. Dependensi ini valid dan berjalan, namun belum terdokumentasi di System Logic UC-006.

## Audit 4 — Tumpang Tindih Kode / Logika Antar Fungsi

### 4.1 Tidak Ada Duplikasi Fungsi (Clean)

- `cn()` — satu sumber di `src/lib/utils.ts`; `src/lib/ui/cn.ts` adalah re-export murni (backward compat). ✅
- `formatCurrency()`, `formatDate()` — hanya di `src/lib/utils.ts`; `mock-data.ts` sudah tidak mendefinisikan sendiri. ✅
- `ProjectStatus` type — hanya dari `src/lib/contracts/enums.ts`. ✅

### 4.2 Potensi Masalah Future — `recalculateProjectProgress` Tidak Ter-expose

Fungsi ini adalah private internal di `wbs-service.ts`. Jika di masa depan ada kebutuhan recalculate dari konteks lain (misalnya saat delete WBS item langsung dari halaman lain), developer baru berpotensi menulis logika yang sama alih-alih meng-export fungsi yang sudah ada.

### 4.3 Denormalisasi Tanpa Update Cascade

`materialUsages.wbsItemName` dan `projectPhotos.wbsItemName` adalah kolom denormalisasi yang menyimpan nama WBS item sebagai string. By design karena tidak ada FK constraint, tetapi perlu dicatat: jika nama WBS item diubah, kolom-kolom ini tidak akan sinkron otomatis.

## Audit 5 — Hal yang Tidak Digunakan / Sudah Tidak Diperlukan

### 5.1 Zustand Store — Dibuat, Belum Digunakan

`src/lib/store/ui-store.ts` dan `src/lib/store/index.ts` tidak dikonsumsi oleh komponen manapun. Saat ini semua state UI dikelola secara lokal. Bukan dead code — ini adalah infrastruktur yang disiapkan, tetapi tidak ada use case konkret yang menggunakannya saat ini.

### 5.2 `src/lib/services/mock-services.ts` — Perlu Verifikasi

Berisi `mockOtpService`. Perlu dicek apakah masih diimpor di mana pun di production path, atau sudah menjadi dead code sejak migrasi ke `email-otp-service.ts`.

### 5.3 `src/lib/ui/cn.ts` — Backward Compat Re-export

Re-export dari `src/lib/utils.ts`. Tidak harus dihapus sekarang, tetapi merupakan kandidat cleanup jika semua import sudah dimigrasikan ke `@/lib/utils`.

### 5.4 `src/lib/services/contracts.ts` — Interface Tidak Terimplementasi (PLANNED)

`FileStorage`, `PDFService`, `PaymentService`, `NotificationService` didefinisikan tapi tidak ada implementasi nyata. Ini bukan dead code — statusnya PLANNED. Perlu dicatat agar tidak dibuat parallel interface di tempat lain ketika implementasi dimulai.

## Catatan Arsitektur

Secara keseluruhan, arsitektur kodebase **solid dan konsisten**. Pola `schemas → service → actions → page` diterapkan secara merata di semua 8 fitur. Tidak ada logika duplikat yang berarti. Isu utama yang ditemukan adalah **drift antara implementasi dan dokumentasi Chain of Truth** — sesuatu yang wajar terjadi ketika development bergerak cepat. Prioritas sesegera mungkin adalah menyinkronkan `data_model.md` dan `user_flows/index.md` agar sesi AI berikutnya tidak beroperasi berdasarkan konteks yang sudah basi.
